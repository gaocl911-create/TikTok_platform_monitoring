## 2. 实施原则

### 2.1 核心业务优先级

一期研发优先级：

```text
用户与数据权限
→ 单作品识别和监控
→ 定时刷新作品指标
→ 正式员工作者新作品发现
→ 运营后台
→ 兼职小程序和任务
→ 审核、佣金与结算
→ 数据迁移和上线
```

作品监控链路未稳定前，不进入大规模兼职任务运营。

### 2.2 平台实体与业务关系分离

重构时必须先落实以下模型：

```text
creator_account       平台作者
content_post          平台作品
monitor_target        谁在监控什么
monitor_target_content 作者监控发现了哪些作品
```

禁止继续使用“一个作者记录同时表达平台作者、数据所有人和监控模式”的结构。

必须保持两条不可变业务规则：

- 作品链接添加后永远只监控该作品，不自动扩展到作者其他作品。
- 作者监控只自动加入监控基线之后发布的新作品，不导入历史作品。

### 2.3 先抖音后小红书

- 一期只实现抖音完整闭环。
- 数据模型保留 `platform` 字段。
- TikHub客户端使用平台适配器接口。
- 小红书开发放在抖音正式上线稳定后。
- 管理后台菜单按平台分开，不提前制作不可用的小红书业务页面。

### 2.4 模块化单体

一期不拆微服务，使用一个RuoYi-Vue-Plus后端工程，通过业务模块隔离职责。

优点：

- 开发和部署成本低。
- 事务边界清晰。
- 便于复用若依用户和权限。
- 适合当前团队和业务规模。
- 后续确有容量需求时再拆采集服务。

### 2.5 异步采集

所有TikHub调用必须通过业务服务或SnailJob异步执行，不允许普通列表接口打开页面时直接调用TikHub。

### 2.6 每阶段可验收

任何阶段结束时都必须形成可运行、可测试、可演示的结果，不以“代码写完”作为完成标准。

## 3. 团队与周期假设

### 3.1 推荐团队

| 角色 | 建议人数 | 主要职责 |
| --- | ---: | --- |
| 产品负责人 | 1 | 需求确认、验收、业务规则 |
| Java后端 | 1至2 | 若依、权限、业务、TikHub、任务调度 |
| 管理后台前端 | 1 | plus-ui业务页面 |
| UniApp小程序 | 1 | 微信登录、任务、作品、佣金 |
| 测试 | 1 | 测试用例、回归、上线验收 |
| 运维 | 兼职投入 | 环境、部署、监控、备份 |

小团队可以由前端兼顾UniApp、后端兼顾部署，但整体周期会增加。

### 3.2 预计周期

| 团队配置 | 一期预计周期 |
| --- | --- |
| 1名全栈开发 | 24至32周 |
| 1后端 + 1前端 | 18至24周 |
| 2后端 + 1后台前端 + 1小程序 + 1测试 | 14至18周 |

周期不包含微信主体认证、隐私合规审核、第三方服务审批等外部等待时间。

### 3.3 推荐基准排期

本计划按4人左右研发团队估算，总周期约16至20周。

部分前后端工作可以并行，但阶段验收依赖不能跳过。

## 5. 阶段0：立项与规则冻结

### 5.1 阶段目标

在创建Java工程前，冻结会影响数据库和核心流程的业务规则。

### 5.2 前置输入

- 重构版PRD。
- 当前FastAPI + Vue 3系统。
- 当前TikHub账号和可用接口。
- 兼职任务原始PRD。
- 微信小程序主体信息。

### 5.3 需要确认的业务规则

必须确定：

1. 兼职是否允许重复领取同一任务。
2. 同一作品是否允许不同兼职重复提交。
3. 兼职入驻是否需要人工审核。
4. 任务佣金是否一期仅使用固定单价。
5. 默认结算周期。
6. 审核驳回后是否继续监控作品。
7. 上级对下级数据是只读还是可以修改、采集和删除。
8. AI文案服务商。
9. 生产文件存储使用MinIO还是OSS。
10. 一期预计用户数、作者数、作品数和每日采集次数。

### 5.4 技术验证

- 确认RuoYi-Vue-Plus稳定版本。
- 确认JDK、Node.js、MySQL和Redis版本。
- 验证SnailJob本地运行。
- 用Java最小程序请求TikHub。
- 测量TikHub关键接口实际响应时间和字段。
- 确认TikHub价格、限流和批量上限。
- 验证微信小程序登录所需主体和配置。

### 5.5 原型范围

至少完成低保真原型：

- 监控作者列表。
- 内容动态。
- 添加作者。
- 添加作品链接。
- 作品详情。
- 采集运行。
- 任务列表。
- 任务提交审核。
- 佣金台账。
- 小程序任务广场。
- 小程序我的任务。
- 小程序提交作品。

### 5.6 交付物

- 冻结版PRD。
- 页面清单和低保真原型。
- TikHub接口字段报告。
- 数据迁移范围说明。
- 一期验收清单。
- 项目排期和负责人。

### 5.7 验收标准

- 不再存在影响核心表结构的重大未决规则。
- TikHub关键接口经过Java真实调用验证。
- 微信登录方案和小程序主体明确。
- 产品、开发、测试对一期范围达成一致。

## 9. 阶段4：指标采集与调度中心

### 9.1 阶段目标

让已添加作品按配置周期自动更新，形成可追溯的历史数据。

### 9.2 数据库任务

创建：

```text
content_snapshot
collection_job
collection_run
alert_record
```

推荐索引：

```text
content_snapshot(content_id, captured_at)
monitor_target(monitoring_status, next_collect_at)
collection_run(status, started_at)
provider_api_call(provider, endpoint, created_time)
```

### 9.3 SnailJob任务

实现：

```text
ScanDueContentTargetsJob
CollectContentMetricsJob
RetryFailedCollectionJob
MarkStaleContentJob
AggregateProviderCostJob
```

扫描任务只查找到期目标并分发工作，不在单次扫描中串行采集全部作品。

### 9.4 Redis锁

锁键建议：

```text
monitor:collect:content:{contentId}
monitor:collect:target:{monitorTargetId}
```

要求：

- 获取失败则标记跳过，不重复调用。
- 锁设置过期时间。
- 任务结束主动释放。
- 只释放当前任务持有的锁。

### 9.5 批量接口

TikHub支持批量统计时：

- 将到期作品按接口批量上限分组。
- 同一批最多使用平台允许数量。
- 记录批次与作品的对应关系。
- 单个作品解析失败不能丢弃整批其他结果。

### 9.6 快照和增长

保存：

- 点赞。
- 评论。
- 收藏。
- 分享。
- 采集时间。
- 数据完整状态。
- Provider调用记录。

计算：

- 单次增长。
- 1小时增长。
- 24小时增长。
- 7天增长。
- 监控以来累计增长。

一期可以查询时计算短期增长；高频统计性能不足后再增加汇总表。

### 9.7 手动采集

接口：

```text
POST /monitor/douyin/content/{id}/collect
POST /monitor/douyin/content/batch-collect
GET  /monitor/collection/run/{runId}
GET  /monitor/collection/run/list
```

行为：

- 立即返回任务ID。
- 前端轮询或通过SSE查询结果。
- 重复点击复用运行中的任务。
- 手动任务也执行锁、预算、限流和重试。

### 9.8 管理后台

- 每条作品“立即采集”按钮。
- 有权限的批量采集。
- 刷新列表按钮与立即采集明确分开。
- 最近采集时间。
- 下次采集时间。
- 单次增长。
- 采集运行列表。
- 失败原因。
- 手动重跑。

### 9.9 测试

- 到期任务会执行，未到期不会执行。
- 暂停目标不会采集。
- 同一作品并发任务只有一个调用成功。
- TikHub超时自动重试。
- 参数错误不重试。
- 批量结果部分失败时成功作品正常保存。
- 快照不串作品。
- 时间和增长量计算正确。
- 预算耗尽后停止调用。

### 9.10 验收标准

- 每30分钟按预期生成快照。
- 没有重复采集风暴。

1. 新系统只读展示迁移数据。

6. 开放少量兼职测试。
7. 扩大到全部用户。
8. 旧系统进入只读。

### 15.4 回滚方案

- 回滚到旧系统的触发条件。

- 数据库迁移回滚方式。
- 小程序版本回退方式。
- TikHub调度总开关。
- 批量采集总开关。

首周重点关注：

- 登录成功率。

- TikHub成功率。

### 15.6 验收标准

## 18. 数据库与迁移规范

### 18.1 表设计

- 所有金额使用 `DECIMAL`，禁止浮点数。
- 平台计数使用 `BIGINT`。
- 平台ID使用字符串保存。
- 所有时间统一UTC。
- 重要JSON保留明确版本。
- 高频查询建立联合索引。
- 唯一约束承担最终幂等保护。

### 18.2 数据库变更

建议使用Flyway管理业务数据库迁移：

```text
V1__init_monitor_tables.sql
V2__init_collection_tables.sql
V3__init_task_tables.sql
V4__init_commission_tables.sql
```

禁止仅通过手工执行SQL修改生产库。

### 18.3 兼容发布

数据库变更采用：

```text
先加字段或新表
→ 部署兼容新旧结构的代码
→ 回填数据
→ 切换读取
→ 稳定后清理旧字段
```

不要在同一次发布中直接删除仍被旧版本使用的字段。

## 19. 接口开发规范

### 19.1 分层

```text
Controller：参数、权限、响应
Application Service：用例编排
Domain Service：核心规则
Repository/Mapper：数据访问
Provider Client：TikHub、微信、AI等外部服务
Job Handler：调度入口
```

Controller不得直接调用Mapper或拼接复杂业务。

### 19.2 幂等

以下接口必须幂等：

- 添加作品。
- 添加作者。
- 领取任务。
- 提交作品。
- 立即采集。
- 审核通过。
- 生成佣金。
- 确认结算。

### 19.3 错误码

按业务域定义：

```text
MONITOR_*
COLLECTOR_*
TASK_*
COMMISSION_*
MINIAPP_*
PERMISSION_*
```

前端必须展示用户可理解的错误，不直接显示Java异常。

### 19.4 API版本

一期内部API可使用：

```text
/api/monitor/*
/api/task/*
/api/commission/*
/api/miniapp/*
```

如果后续对外开放，再增加明确的 `/open-api/v1`。

## 22. 主要风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| TikHub字段或接口变化 | 采集失败 | Provider适配层、契约测试、原始响应版本 |
| TikHub费用增长 | 成本失控 | 批量接口、分级频率、预算、调用统计 |
| 新作品发现延迟 | 业务漏监控 | 独立发现任务、积压监控、人工立即检查 |
| 作者历史作品误加入 | 数据污染 | 基线时间 + 已见作品ID双重判断 |
| 单作品变成整号监控 | 成本和范围错误 | 独立monitor_target，严格按target_type调度 |
| 数据权限越权 | 隐私和业务风险 | 后端资源级校验、自动化越权测试 |
| 任务名额超发 | 财务风险 | 事务、行锁、幂等键 |
| 重复生成佣金 | 财务损失 | 提交唯一佣金约束、幂等服务 |
| 小程序审核等待 | 上线延迟 | 阶段0提前准备主体、隐私协议和类目 |
| 旧数据时区混乱 | 发布时间错误 | 抽样验证、迁移映射、UTC规范 |
| 新旧系统重复采集 | API费用重复 | 灰度账号互斥、旧系统只读切换 |

## 25. 计划完成定义

本实施计划执行完成需要满足：

- PRD一期全部P0流程实现。
- 抖音作者和单作品两种监控模式严格分离。
- 正式员工作者只发现添加后的新作品。
- 兼职人员任务作品和独立作品均只按单作品监控。
- 作品指标每30分钟可配置采集。
- 作者主页每6小时可配置刷新。
- 上级、部门负责人和管理员数据权限正确。
- 微信小程序完成登录、入驻、任务、提交和个人数据查看。
- 后台完成审核、佣金和人工结算。
- TikHub调用和成本可追溯。
- 旧系统有效数据迁移并通过对账。
- 生产环境具备监控、备份、灰度和回滚能力。
